# Pipeline: как работает audio2text

## Общая схема

```
АУДИОФАЙЛ (.opus)
    |
    v
[1] ТРАНСКРИБАЦИЯ (MLX Whisper)          ~90 сек / 60 мин аудио
    |   whisper-large-v3-turbo
    |   -> .txt, _timed.txt
    v
[2] ДИАРИЗАЦИЯ (pyannote 3.1)           ~1000 сек / 60 мин аудио
    |   speaker-diarization-3.1
    |   -> _diar.txt
    v
[3] СУММАРИЗАЦИЯ (Qwen2.5-14B-4bit)     ~200 сек / 60 мин аудио
    |   mlx-lm, chunked если > 19K токенов
    |   -> _summary.txt
    v
[4] OBSIDIAN EXPORT (опционально)        < 1 сек
        -> {дата} {название}.md
```

---

## Шаг 1: Транскрибация

**Файл:** `transcriber.py` -> `MLXTranscriber`
**Модель:** `mlx-community/whisper-large-v3-turbo` (~2.3 GB)

### Как работает Whisper

1. **Загрузка аудио** — ffmpeg декодирует .opus в raw PCM, ресемплит до 16 kHz моно
2. **Разбивка на окна** — аудио нарезается на 30-секундные окна (стандарт Whisper)
3. **Mel-спектрограмма** — каждое окно преобразуется в 80-канальную мел-спектрограмму (128 мел-полос для large-v3)
4. **Encoder** — трансформер-энкодер обрабатывает спектрограмму, создает эмбеддинги
5. **Decoder** — авторегрессивный декодер генерирует текст токен за токеном
6. **Word timestamps** — дополнительный проход через cross-attention для привязки слов к таймкодам

### Параметры

| Параметр | Значение | Влияние |
|----------|----------|---------|
| `language` | `"ru"` | Пропускает автодетекцию языка, экономит время |
| `word_timestamps` | `True` | Точные таймкоды слов, +10-15% времени |
| `beam_size` | `1` (MLX ограничение) | Greedy decoding, нет beam search |

### Выход

```
segments: [
  {"start": 0.0,  "end": 3.5,  "text": "Привет, коллеги"},
  {"start": 3.5,  "end": 7.2,  "text": "Давайте начнем"},
  ...
]
```

Сохраняется в два файла:
- `.txt` — чистый текст
- `_timed.txt` — с таймкодами `[00:00-00:03] Привет, коллеги`

### Узкие места

- **GPU memory**: модель ~2.3 GB в VRAM (Metal shared memory)
- **beam_size=1**: MLX Whisper не поддерживает beam search, качество чуть ниже чем faster-whisper с beam_size=5
- **Длинные файлы**: обрабатываются последовательно по 30 сек, нет параллелизма

### Как улучшить качество

- **Модель покрупнее**: `whisper-large-v3-mlx` (full, не turbo) — +1% WER, но медленнее
- **faster-whisper на CPU/CUDA**: поддерживает `beam_size=5`, точнее на сложном аудио
- **Пре-процессинг аудио**: нормализация громкости, шумоподавление перед транскрибацией

### Как снизить ресурсы

- **Модель полегче**: `whisper-large-v3-turbo-q4` (~0.6 GB, 4-bit квантизация)
- **Язык явно**: `language: "ru"` уже стоит, пропускает автодетекцию

## Шаг 2: Диаризация

**Файл:** `diarizer.py` -> `Diarizer`
**Модель:** `pyannote/speaker-diarization-3.1`

### Как работает pyannote pipeline

Внутри pipeline 4 последовательных этапа:

```
АУДИО (waveform, 16kHz)
    |
    v
[2.1] Voice Activity Detection (VAD)
    |   Сегментационная модель — находит где речь, где тишина
    |   Выход: временные интервалы с речью
    v
[2.2] Speaker Embedding Extraction
    |   Для каждого речевого сегмента извлекает вектор-эмбеддинг
    |   говорящего (модель типа ECAPA-TDNN или WavLM)
    |   Это САМЫЙ ДОЛГИЙ этап — обрабатывает каждый чанк отдельно
    v
[2.3] Clustering
    |   Агломеративная кластеризация эмбеддингов
    |   Определяет сколько спикеров и группирует сегменты
    v
[2.4] Speaker Assignment
        Присваивает метки SPEAKER_00, SPEAKER_01, ... каждому сегменту
```

### Слияние с транскрибацией

```python
# Для каждого сегмента транскрибации:
#   находим спикера с максимальным перекрытием по времени
overlap = max(0, min(seg_end, turn_end) - max(seg_start, turn_start))
speaker = argmax(overlap)
```

### Параметры

| Параметр | Значение | Влияние |
|----------|----------|---------|
| `device` | `"cpu"` | MPS нестабилен для pyannote |
| `min_speakers` | `null` (авто) | Можно задать если известно заранее |
| `max_speakers` | `null` (авто) | Ограничивает кластеризацию |
| `hf_token` | обязателен | Gated model на HuggingFace |

### Выход

```
_diar.txt:
[00:00-00:03] SPEAKER_00: Привет, коллеги
[00:03-00:07] SPEAKER_01: Давайте начнем
```

### Узкие места

- **Самый медленный шаг всего pipeline** (~1000 сек на 60 мин аудио)
- **Embedding extraction**: O(n) по количеству речевых сегментов, каждый проходит через нейросеть
- **CPU-only**: MPS (Metal) нестабилен, CUDA недоступен на Mac
- **Память**: pyannote + torch загружают ~3-4 GB RAM

### Как улучшить качество

- **Задать min/max_speakers**: если известно количество участников, убирает ошибки кластеризации
- **Качество аудио**: чем чище запись, тем точнее разделение спикеров
- **Отдельные микрофоны**: если у каждого участника свой микрофон, диаризация не нужна

### Как снизить ресурсы

- **Пропустить если не нужно**: `diarization.enabled: false` — экономит ~80% времени pipeline
- **min/max_speakers**: ускоряет кластеризацию, меньше итераций
- **Нет хороших легких альтернатив**: pyannote 3.1 — де-факто стандарт

## Шаг 3: Суммаризация

**Файл:** `summariser.py` -> `LLMSummarizer`
**Модель:** `mlx-community/Qwen2.5-14B-Instruct-4bit` (~8.5 GB)

### Пре-процессинг входного текста

```python
# 1. Убираем таймкоды
"[00:01-00:06] SPEAKER_02: текст" -> "Спикер 2: текст"

# 2. Дедупликация повторов (пока пока x100 -> x7 max)
#    Одна реплика на спикера в группе одинаковых фраз
```

### Обычный режим (текст < 19404 токена)

```
Промпт (инструкция + контекст + транскрипция)
    |
    v
Qwen2.5-14B generate(max_tokens=4096)
    |
    v
Пост-процессинг (удаление повторов в выводе)
    |
    v
_summary.txt
```

### Chunked режим (текст > 19404 токена)

```
Длинный текст
    |
    v
Разбивка на чанки (с 10-строчным перекрытием)
    |
    v
Чанк 1 -> summarize -> Частичное резюме 1
Чанк 2 -> summarize -> Частичное резюме 2
Чанк N -> summarize -> Частичное резюме N
    |
    v
Финальный проход: "Объедини N резюме в одно"
    |
    v
_summary.txt
```

### Бюджет токенов

```
Контекст модели:         32768 токенов (Qwen2.5)
Консервативный лимит:    24000 токенов
  - Промпт/инструкция:     500 токенов
  - Выход (max_tokens):    4096 токенов
  = Входной текст макс:   19404 токенов

1 токен ~ 2-3 символа русского текста
19404 токена ~ 40-60K символов ~ 30-45 мин встречи
```

### Параметры

| Параметр | Значение | Влияние |
|----------|----------|---------|
| `max_tokens` | `4096` | Максимум выходного текста |
| `temperature` | не задан (default) | mlx-lm default = 0.0 (greedy) |
| `repetition_penalty` | **не поддерживается** | mlx-lm не имеет этого параметра |
| `task` | `"summarize"` | Определяет промпт |
| `context` | `"IT-терминология"` | Добавляется в промпт |

### Узкие места

- **Память**: Qwen 14B-4bit занимает ~8.5 GB shared memory
- **Нет repetition_penalty**: модель может зацикливаться, спасает только пост-процессинг
- **Chunked = N+1 вызовов LLM**: 3 чанка = 4 вызова (3 + merge), каждый ~60-200 сек
- **Greedy decoding**: без beam search и sampling

### Как улучшить качество

- **API вместо локальной модели**: Claude/GPT-4o значительно качественнее для суммаризации
- **Модель побольше**: `Qwen2.5-32B-Instruct-4bit` если хватает RAM (~18 GB)
- **Лучший промпт**: добавить примеры (few-shot), уточнить контекст
- **Diar вместо timed**: уже используется — спикеры дают контекст кто что сказал

### Как снизить ресурсы

- **Модель полегче**: `Qwen2.5-7B-Instruct-4bit` (~4.5 GB) — быстрее, но хуже качество
- **API**: не грузит локальную память, но нужен интернет и платный ключ
- **Пропустить если не нужно**: можно работать только с транскрибацией

## Шаг 4: Obsidian Export

**Файл:** `processor.py` -> `export_obsidian_note()`

### Как работает

1. Парсит имя файла: `2026-04-10 16-04 Название.opus` -> дата, время, название
2. Считает спикеров из `_diar.txt`
3. Генерирует Markdown с YAML frontmatter:

```markdown
---
date: 2026-04-10
time: "16:04"
type: meeting
speakers: 6
source: 2026-04-10 16-04 Название.opus
tags: [meeting, audio2text]
---

# Название

{содержимое _summary.txt}
```

4. Сохраняет в `{vault_path}/{folder}/{date} {title}.md`

### Ресурсы: пренебрежимо малы

## Общие рекомендации по оптимизации

### Приоритеты по времени (60 мин аудио)

| Шаг | Время | % от pipeline |
|-----|-------|---------------|
| Диаризация | ~1000 сек | **76%** |
| Суммаризация | ~200 сек | 15% |
| Транскрибация | ~90 сек | 7% |
| Export | <1 сек | <1% |

### Максимум качества (текущая конфигурация подходит)

1. whisper-large-v3-turbo — хороший баланс скорость/качество
2. pyannote 3.1 — лучшая open-source диаризация
3. Qwen2.5-14B — неплохо, но **API-модель (Claude/GPT-4o) будет заметно лучше** для саммари

### Максимум скорости

1. Пропустить диаризацию если не нужны спикеры (-76% времени)
2. whisper-large-v3-turbo-q4 для транскрибации (-30% времени)
3. Qwen2.5-7B-4bit для суммаризации (-50% времени, -4 GB RAM)
4. `HF_HUB_OFFLINE=1` — убрать сетевые проверки при загрузке моделей

### Управление памятью (автоматическая выгрузка)

В pipeline каждая модель выгружается после своего шага (`unload=True`):

```
[1] Whisper загружен (+2.3 GB) -> транскрибация -> выгружен (-2.3 GB)
[2] pyannote загружен (+3-4 GB) -> диаризация -> выгружен (-3-4 GB)
[3] Qwen загружен (+8.5 GB) -> суммаризация -> выгружен (-8.5 GB)
```

**Peak memory**: ~8.5 GB (только одна модель в памяти одновременно)
**Без выгрузки было бы**: ~14 GB (все три модели одновременно)

Выгрузка включает:
- Удаление ссылок на модель + `gc.collect()`
- Очистка Metal кэша (`mx.metal.clear_cache()`) для MLX моделей
- Очистка CUDA кэша (`torch.cuda.empty_cache()`) для GPU моделей
- Удаление кэшированных модулей mlx_whisper из `sys.modules`

При отдельном вызове (не pipeline) модели не выгружаются — `unload=False` по умолчанию.
